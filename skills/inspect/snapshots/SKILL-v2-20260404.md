# Accessibility Audit

Conduct a structured accessibility audit and produce a report. This is a thorough, systematic review — not a quick scan.

## Process

### Step 1: Define Scope

Ask the user (if not specified):

1. **Scope**: Entire project / specific page / specific component?
2. **Target level**: WCAG 2.2 AA (default) or AAA?
3. **Target jurisdictions**: Which laws apply? (US ADA, EU EAA, Japan JIS, Taiwan, Canada AODA, Australia DDA)
4. **Platform**: Web / iOS / Android / Desktop / Cross-platform?
5. **Existing tooling**: Is axe-core, Lighthouse, eslint-plugin-jsx-a11y installed?

### Step 2: Automated Scan (if applicable)

If the project has automated tools available, run them first:

```bash
# axe-core via Playwright
npx playwright test --grep accessibility

# Lighthouse CLI
npx lighthouse <url> --only-categories=accessibility --output=json

# eslint a11y plugin
npx eslint --rule 'jsx-a11y/*' src/
```

Document: automated tools catch ~30-40% of issues. The remaining 60-70% requires manual review.

### Step 3: Manual Review

Check each category systematically. Read `../a11y-advisor/references/wcag-quick.md` for criterion details.

#### 3a. Color & Contrast
- [ ] Text contrast ≥4.5:1 (AA) or ≥7:1 (AAA)
- [ ] Large text contrast ≥3:1 (AA) or ≥4.5:1 (AAA)
- [ ] UI component contrast ≥3:1
- [ ] Information not conveyed by color alone
- [ ] Test with simulated color vision deficiency
- [ ] **Cross-test: `prefers-contrast: more` + dark mode** — high-contrast CSS variables must not create dark-on-dark or light-on-light when combined with theme toggle (Android Bold text triggers this)
- [ ] **CSS variables in dark mode** — verify `color: var(--x)` resolves correctly in both themes. Avoid `color: white` — use `color: var(--bg)` for auto-adaptation

#### 3b. Keyboard Navigation
- [ ] All interactive elements reachable via Tab
- [ ] Focus order is logical (matches visual order)
- [ ] Focus indicator always visible (`:focus-visible`)
- [ ] No keyboard traps
- [ ] Focused element not obscured by sticky headers/footers (2.4.11 NEW)
- [ ] Modal dialogs trap focus correctly, Escape closes
- [ ] Skip links or landmark navigation present

#### 3c. Screen Reader
- [ ] Page has descriptive `<title>`
- [ ] Landmarks present (`<header>`, `<nav>`, `<main>`, `<footer>`)
- [ ] Heading hierarchy correct (no skipped levels)
- [ ] All images have appropriate alt text
- [ ] All form inputs have associated labels
- [ ] All buttons have accessible names
- [ ] Dynamic content updates announced (aria-live)
- [ ] Language attribute set on `<html>`
- [ ] No Taiwan `:::` navigation markers (導盲磚) — never complied with WCAG, removed from 2017 TW standard

#### 3d. Forms
- [ ] Every input has visible, associated `<label>`
- [ ] Required fields indicated (not by color alone)
- [ ] Error messages identify the field and suggest correction
- [ ] Errors announced to screen readers (role="alert" or aria-live)
- [ ] No redundant entry required (3.3.7 NEW)
- [ ] Authentication doesn't require cognitive function tests (3.3.8 NEW)
- [ ] `autocomplete` on personal data fields

#### 3e. Media
- [ ] Video has captions (prerecorded)
- [ ] Audio has transcript
- [ ] Auto-playing media can be paused/stopped
- [ ] No content flashes >3 times/second (LIFE-SAFETY)

#### 3f. Motion & Animation
- [ ] `prefers-reduced-motion` respected
- [ ] `prefers-reduced-transparency` respected
- [ ] Time limits adjustable or removable
- [ ] Auto-moving content can be paused
- [ ] `transition: all` not used with CSS custom properties (causes visual lag during theme switches — use specific properties instead)

#### 3g. Touch & Target Size
- [ ] Interactive targets ≥24×24 CSS pixels (2.5.8 NEW)
- [ ] Drag operations have single-pointer alternative (2.5.7 NEW)
- [ ] Multi-point gestures have single-pointer alternative
- [ ] Content not restricted to single orientation

#### 3h. Cognitive
- [ ] Consistent navigation across pages
- [ ] Consistent identification (same function = same label)
- [ ] Help mechanism in consistent position (3.2.6 NEW)
- [ ] Clear, simple language where possible
- [ ] No dark patterns or guilt-inducing design
- [ ] Undo/confirmation for destructive actions

#### 3i. Agent Operability (WCAG 3.0 direction)
- [ ] Accessibility tree properly exposes all interactive elements
- [ ] Semantic HTML used (not div-based UI)
- [ ] ARIA states reflect actual component state
- [ ] No hover-only content (invisible to non-mouse agents)

### Step 5: Generate Accessibility Statement

Offer to generate an **accessibility statement** for the audited site/app. This is the international standard replacement for static accessibility badges. Taiwan's badge system is an outdated 1990s model; advanced nations transitioned to dynamic statements around 2010.

Reference: W3C WAI — Developing an Accessibility Statement (https://www.w3.org/WAI/planning/statements/)

Template:

```markdown
# Accessibility Statement

[Organization name] is committed to ensuring digital accessibility for people
with disabilities. We are continually improving the user experience for
everyone and applying the relevant accessibility standards.

## Standards Applied

This [website/application] aims to conform to **WCAG 2.2 Level AA**.

## Known Limitations

Despite our efforts, some content may not yet be fully accessible:

| Area | Issue | Expected Resolution |
|------|-------|---------------------|
| [component/page] | [brief description of barrier] | [target date or "under investigation"] |

## Feedback

We welcome your feedback on the accessibility of [site name].
Please let us know if you encounter accessibility barriers:

- Email: [accessibility contact]
- Phone: [number]
- Postal: [address]

We try to respond to feedback within [X] business days.

## Compatibility

This site is designed to be compatible with:
- Recent versions of screen readers (NVDA, JAWS, VoiceOver, TalkBack)
- Recent versions of major browsers (Chrome, Firefox, Safari, Edge)

## Assessment

This statement was last reviewed on [date] based on:
- Automated testing with [axe-core / Lighthouse]
- Manual review against WCAG 2.2 AA criteria
- [Screen reader testing with NVDA/VoiceOver] (if applicable)
- [User testing with assistive technology users] (if applicable)

## Applicable Laws

[List jurisdictions from audit scope: US ADA, EU EAA, Japan JIS, Taiwan, etc.]
```

Key points when generating:
- **Known Limitations must be honest.** Populate from the audit's Critical and Warning findings. Jedi Lin documents that Taiwan's 數發部 rejected accessibility statements because officials "couldn't understand" how agencies could publicly acknowledge barriers. This transparency IS the point.
- **Expected Resolution dates must be realistic.** Use the remediation priority (P0/P1/P2) to set dates.
- **The statement is a living document.** Suggest updating it after each re-audit.
- **For Taiwan government projects**: Explain that this replaces the outdated 標章 (badge) system. The badge is a static snapshot; the statement is a continuous commitment.

### Step 6: Follow Up

After delivering the report and accessibility statement:
- Offer to create OpenSpec scenarios for critical findings (if user uses OpenSpec)
- Offer to create GitHub issues / todo items for each finding
- Suggest re-audit timeline based on severity
- Offer to generate a public accessibility statement (Step 5)
